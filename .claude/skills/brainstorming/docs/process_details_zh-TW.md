# 腦力激盪流程細節與核心原則

## 流程步驟

### 理解想法
- 首先檢視目前的專案狀態（檔案、文件、近期的 commits）。
- 一次提問一個問題來完善這個想法。
- 盡可能優先給予選擇題，但開放式問答也可以。
- 每次回覆只問一個問題——如果某個主題需要深入探討，請拆分成多個問題來問。
- 專注於理解：目的、限制、成功標準。

### 探索作法
- 提出 2-3 種不同的作法，並附上各自的優缺點。
- 以對話的方式呈現這些選項，並說明你的推薦與理由。
- 將你推薦的選項放在最前面，並解釋為什麼。

### 呈現設計
- 一旦你認為已經了解要建置什麼，就開始呈現設計內容。
- 根據複雜度調整每個段落的長度：如果是直觀的功能只需幾句話，如果是較複雜的細節則可寫 200-300 字。
- 在呈現完每個段落後，詢問目前看起來是否符合預期。
- 內容應涵蓋：架構、元件、資料流、錯誤處理、測試。
- 如果有不合理的地方，請隨時準備回頭重新釐清。

## 設計完成後

### 文件紀錄
- 將確認後的設計記錄至 `docs/plans/YYYY-MM-DD-<topic>-design.md`。
- 如果有這個 skill，可以使用 `elements-of-style:writing-clearly-and-concisely` 進行優化。
- 將設計文件 commit 到 git 中。

### 實作
- 呼叫 `writing-plans` skill 來建立詳細的實作計畫。
- **絕對不要**呼叫任何其他 skill。下一步就只有 `writing-plans`。

## 核心原則

- **一次問一個問題** - 不要用大量問題淹沒使用者。
- **傾向選擇題** - 在可能的情況下，選擇題會比開放式問題更容易回答。
- **果斷地遵守 YAGNI 原則** - 從所有設計中移除不必要的功能。
- **探索替代方案** - 在定案之前，永遠先提出 2-3 種作法。
- **漸進式驗證** - 提出設計，在繼續下一步之前先取得核准。
- **保持彈性** - 當有不合理的地方時，隨時回去釐清。
